## Resolved items

- **SKILLS-MOUNT-2** — _Resolved 2026-05-11 immediately after SKILLS-MOUNT-1._ After SKILLS-MOUNT-1 the in-container `/skills` endpoint reported `settings.json` missing and zero user-level skills/agents/commands — yet the same paths on the host had content. Investigation via `container exec agent-host-cc ls -la /home/node/.claude` revealed the host's `~/.claude` is largely a set of **symlinks pointing to absolute paths outside `~/.claude`** (in this case `/Users/giorgosmarinos/claude-workdocs/.claude/{skills,agents,commands,settings.json,scripts,…}`). The single bind-mount only carried the symlinks themselves into the container; the symlink targets (absolute host paths) did not exist inside Alpine, so every symlink dangled. The `plugins/` directory was the only top-level entry that worked because it is a real directory, not a symlink. **Fix:** added `discover_claude_extra_mounts()` to `scripts/run-docker.sh` and `scripts/run-apple-container.sh`. It walks the top level of `${CLAUDE_DIR}`, resolves each absolute-path symlink whose target escapes the mount, deduplicates by parent directory, and emits each as an additional `-v <abs>:<abs>:${CLAUDE_MODE}` bind-mount preserving the absolute path so symlinks resolve inside the container. After redeploy, `/skills` returned `skillCount: 331 (29 user + 302 plugin), agentCount: 14, commandCount: 27, settings.exists: true` — matching the host exactly, no warnings. **Caveat:** only top-level symlinks are auto-detected; nested symlinks under skills/, agents/, etc. would need a deeper walk if any user ever uses them. Recorded so the next operator who has a layout-by-symlinks setup does not rediscover the failure mode from scratch.

- **SKILLS-MOUNT-1** — _Resolved 2026-05-11 during live `~/.claude` mount investigation._ After enabling the `CLAUDE_DIR=$HOME/.claude → /home/node/.claude` bind-mount in `scripts/run-docker.sh` and `scripts/run-apple-container.sh`, the in-container agent reported only ~9 skills (the user-level top-level ones plus a few built-ins) and zero plugin-provided skills, even though the host has 302 plugin skills enabled. Root cause: `src/claudeCodeRunner.ts:247` passed `settingSources: ["project"]` to the Claude Agent SDK's `query()`, which told the SDK to ignore the `user` layer (`~/.claude/settings.json` + `~/.claude/plugins/**`). Consequence: the mount data was readable, but the SDK never enabled plugins because it never read the user settings file. **Fix:** extracted the setting-sources value into an exported constant `CLAUDE_SETTING_SOURCES = ["user", "project", "local"] as const` (`src/claudeCodeRunner.ts`), and pass it both to the SDK and to the new `/skills` debug endpoint so the introspection always matches actual SDK behaviour. **Added `/skills` debug endpoint:** `GET /skills` (Bearer-auth, same key as other endpoints) walks the mounted `~/.claude` filesystem and returns a JSON snapshot of `{claudeDir, exists, settingSources, settings:{path,exists,topLevelKeys}, skills[], agents[], commands[], plugins[], warnings[]}`. Implemented in `src/skillsIntrospector.ts` with `MAX_RECURSION_DEPTH=6` and full error swallowing — the endpoint never throws because of an unreadable subdirectory. Smoke-tested against the host's actual `~/.claude`: returned 331 skills (29 user + 302 plugin), 14 agents, 27 commands, 3 top-level plugin trees. **Security trade-off accepted:** including the `"user"` layer means the in-container agent inherits all the user's hooks, plugin enablement, and any auto-allowed permissions from `~/.claude/settings.json`; combined with `permissionMode: "bypassPermissions"` already on file, this means the container should be treated as fully trusted with the host user's Claude configuration. Documented here so it is not forgotten.

- **CU-RELAY-1** — _Resolved 2026-05-10 during live chat-ui smoke test._ The chat-ui's `POST /api/chat` relay returned `502 upstream_error: This operation was aborted` for every request, even though the upstream `agent-host-cc` host service responded normally to the same payload via `curl`. Root cause: `chat-ui/server/chatRelay.ts:116` registered the disconnect listener as `request.raw.on("close", ...)`. In Node.js 18+ an `IncomingMessage` emits `'close'` once its readable side is consumed (i.e. as soon as Fastify's `application/json` parser finished reading the body), which fired the AbortController immediately and aborted the upstream `undici.request` before it could complete. The integration tests passed because `app.inject()` does not exercise the real socket lifecycle (the test-builder agent had pre-flagged this as "AbortController mid-stream disconnect test is partial — `app.inject()` cannot simulate a mid-stream TCP close"). **Fix:** the listener was moved to `request.raw.socket.on("close", ...)`, with a `detachClose()` helper centralising the four detach sites. The socket's `close` event fires only on actual client disconnect, which is the intended trigger. Live retest via `curl` against `POST /api/chat` now returns `200` with proper SSE chunks and a `[DONE]` terminator. `npx tsc -p tsconfig.server.json --noEmit` exits 0; `npx vitest run` exits 0 with 242/242 passing.

- **CU-WORKSPACE-1** — _Resolved 2026-05-10 during live chat-ui smoke test._ The `agent-host-cc` host container, running under Apple's `container` CLI with the named volume `agent-host-cc-workspace` mounted at `/workspace`, returned `500 internal: EACCES: permission denied, mkdir '/workspace/derived-00000d01'` on the first chat request that needed a per-chat workspace. Root cause: a fresh named volume initialises with `root:root` ownership and overrides the image's pre-`chown`'d `/workspace` directory, but the in-container process runs as `node` (uid=1000) and cannot write into a root-owned directory. The Dockerfile's `RUN mkdir -p /workspace && chown node:node /workspace` only takes effect inside the image layer; the volume mount masks it. ADR-6 / BUILD-1 (already on file) constrain us to keep the `node` user. **Live fix (already applied to the running container 09ca39e7-...):** `container exec --user root <id> chown node:node /workspace`, which the named volume now persists across restarts. **Durable fix (applied to image):** added `scripts/docker-entrypoint.sh` which (a) runs as root (the Dockerfile now sets `USER root` immediately above `ENTRYPOINT`), (b) `chown node:node /workspace` (idempotent), then (c) drops privileges via `su-exec node:node` and execs the requested command. The Dockerfile installs `su-exec` from Alpine's package repo and sets `ENTRYPOINT ["/usr/local/bin/docker-entrypoint.sh"]`. The main process therefore still runs as uid=1000, preserving ADR-6. Image rebuild required to make the durable fix active in newly created containers (`./scripts/build-apple-container.sh`); not blocking — the running container is already correct.

- **DEFECT-1** — _Resolved 2026-05-10 during first live container run._ The Files API config (`FILES_API_BASE_URL`, `FILES_API_KEY`) was required at startup, but the design intent is that it is only needed when a request includes the `files[]` extension. A user running text-only chat completions could not start the container without inventing placeholder values. **Fix:** the two variables are now optional in `src/config.ts`. Partial configuration (only one set) is rejected at startup with `ConfigurationError`. `src/attachmentProcessor.ts` accepts `filesApi: FilesApiOptions | undefined`; when undefined, `files[]` entries are silently dropped (matches the existing swallow-on-fetch-failure semantics). `src/index.ts` logs `files API disabled: …` at INFO when the feature is off. Three new vitest cases assert: optional behaviour, partial-config rejection (BASE_URL set + KEY missing), partial-config rejection (KEY set + BASE_URL missing). Configuration guide §5 and `.env.example` updated to reflect the optional status. Tests: 96/96 pass.
